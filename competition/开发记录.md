## 二、AI 工具使用情况

### 1. 主要使用的 AI 工具
- **Claude Code**（代码生成 + 架构推理）
- **DeepSeek**（系统内置功能验证）
- **智谱清言**（多模型对比测试）

### 2. AI 在不同任务中的角色

#### 代码生成
- 后端 Service 层模板化代码（CRUD）由 AI 生成约 60%
- 前端 Vue 组件骨架由 AI 生成约 50%
- 工具类（如 JwtService、文件解析）由 AI 生成约 70%
- **人工**：业务逻辑、异常处理、边界情况补充

#### 架构设计
- AI 提供初步技术选型建议（如轻量级 RAG 方案 vs 向量数据库）
- AI 协助论证三阶段质量闭环的可行性
- **人工**：最终决策、性能权衡、可维护性把控

#### 提示词工程
- AI 协助撰写教案/题库的初版 Prompt
- AI 协助设计分级模板（基础/标准/详尽/特详）
- **人工**：根据真实教学规范多轮迭代调优

#### Bug 修复
- 把异常堆栈喂给 AI，让 AI 分析根因
- AI 给出修复建议
- **人工**：验证修复方案、防止破坏性修改

## 三、关键技术决策记录

### 决策1：选择 OpenAI 兼容协议接入大模型
- **背景**：DeepSeek、智谱清言、Kimi 都提供 OpenAI 兼容 API
- **决策**：统一使用 OpenAI 协议，通过 Base URL 区分厂商
- **结果**：实现了多模型零成本切换，符合"国产大模型"要求

### 决策2：轻量级 RAG vs 向量数据库
- **背景**：RAG 是核心能力，但向量库（Milvus/Qdrant）部署复杂
- **决策**：使用 TF 关键词匹配 + 智能分块的轻量方案
- **结果**：部署成本降低，适合高职院校 IT 环境，符合"操作简单易用"要求

### 决策3：教案分四个内容等级
- **背景**：不同场景对教案精细度要求不同
- **决策**：基础版（4段）/ 标准版（6段）/ 详尽版（8段）/ 特详版（10段）
- **结果**：覆盖快速备课、日常教学、公开课、教学检查等多场景

### 决策4：质量保障三阶段闭环
- **背景**：大模型输出不稳定，单次生成质量难以保证
- **决策**："生成 → 合规自检 → AI 批判 → 修正重生成" 闭环
- **结果**：教案合格率从 60% 提升到 90% 以上

### 决策5：动态课程上下文提示词工程
- **背景**：每次切换课程都改 Prompt 不现实
- **决策**：提示词模板化 + 课程配置占位符
- **结果**：一次配置，跨课程通用

## 四、AI 生成代码的人工审核要点

为避免 AI 生成代码的潜在问题，建立了以下审核 checklist：

- [ ] 是否处理了 null 值与空集合
- [ ] 是否有 SQL 注入或 XSS 风险
- [ ] 是否有未关闭的资源（流、连接）
- [ ] 异常是否被正确捕获和包装
- [ ] 是否有过时的 API 或不推荐的写法
- [ ] 中文注释和日志是否清晰
- [ ] 是否符合项目的命名规范

## 五、开发过程中的关键 Bug 与解决

### Bug 1：前端 API 路径双重 /api 前缀
- **现象**：访问"课程""知识库""智能答疑"时返回服务器内部错误
- **根因**：新增 API 文件中路径携带 /api 前缀，与 axios baseURL 重复
- **修复**：去掉新 API 文件中路径的 /api 前缀
- **教训**：新增 API 文件应参考现有规范（auth.js）

### Bug 2：DashboardService 中 NPE 风险
- **现象**：仪表盘偶发 500 错误
- **根因**：聚合查询返回的 cnt 字段可能为 null，强转 Number 抛 NPE
- **修复**：增加 null 校验
- **教训**：对外部数据源（DB 聚合结果）必须做 null 防御

### Bug 3：YaHei 字体不支持 Emoji
- **现象**：架构图导出后所有 emoji 显示为方框
- **根因**：matplotlib 使用 SimHei/YaHei 字体，不含彩色 emoji 字形
- **修复**：将 emoji 替换为 Unicode 几何符号（◆ ● ★ ■ ▲ ◇ 等）
- **教训**：导出图选择字体时要考虑字符集覆盖

## 六、第二轮迭代（Phase 1-5 + 多语言增强）

### 1. 改造背景

初版上线后实际试用中暴露 5 类问题：性能（60 份串行 30-60 分钟）、质量（答案聚簇 / 题干重复）、多语言（编程题硬编码 Java）、进度可视（1500ms 轮询 + 18% 跳 100% 跳变）、计费公平性（题库固定 5 分）。

### 2. Phase 1-5 概览（共 51 个原子步骤）

| Phase | 步骤数 | 关键成果 |
| --- | --- | --- |
| Phase 1 基础设施 | 22 | LLM 参数化（ChatOptions 5 档场景）/ 限流框架 / 文件解析缓存 / Prompt 模块化 / 共享基类 |
| Phase 2 性能 | 11 | BM25 检索精筛 / 并行调度器 / 教案题库全并行 / 评审批量化基础设施 |
| Phase 3 质量 | 8 | 答案语义校验 / 分布均衡 / 题干去重 / 编程题运行校验 / 批量评审 service 接入 |
| Phase 4 进度反馈 | 4 | TaskProgressBus + SSE 接口 + TaskService 集成 |
| Phase 5 端到端 | 6 | 编程题样例校验 / SSE 鉴权（?token=）/ 前端 EventSource / 配置项 UI 暴露 |

### 3. 第二轮关键技术决策

#### 决策 6：BM25 vs 向量检索（再次升级）
- **背景**：教案/题库主链路仍用 distributeTruncate 盲截断
- **决策**：用纯 Java BM25 倒排索引（k1=1.5, b=0.75，中英混合 unigram+bigram 分词），不引入第三方依赖
- **结果**：长 PPT 末尾噪声不再喂给 LLM，token 消耗降低约 30-50%

#### 决策 7：并行 vs 串行（4 倍加速）
- **背景**：60 份教案串行 30-60 分钟用户等不起
- **决策**：CompletableFuture + 独立 llmCallExecutor 线程池（core=16/max=32）+ LlmRateLimiter 按 provider 信号量节流
- **结果**：60 份从 30-60 分钟降到 1-2 分钟（4× 加速）

#### 决策 8：编程题多语言（4 段优先级）
- **背景**：硬编码 Java，上传 Python PPT 也生成 Java
- **决策**：4 段优先级（请求层 > 课程配置 > 素材自动检测 > 默认 Java）+ 14 种语言下拉 + Prompt"绝不混用"硬约束 + CodeCompilerRouter 校验路由
- **结果**：彻底解决语言失配；非 Java 语言走轻量代码格式校验

#### 决策 9：SSE 实时进度 vs 长轮询
- **背景**：1500ms 轮询 + 进度条 18% 跳 100%，用户体验差
- **决策**：Spring MVC SseEmitter + TaskProgressBus 进程内 Map + GenerationParallelizer.onProgress 回调每完成一项推送 + 失败回退轮询保底
- **结果**：进度更新延迟 ≤1.5s → < 100ms；进度条匀速 1/N、2/N、3/N 直达 N/N

#### 决策 10：积分两段式计费 vs 固定值
- **背景**：题库无论 1 题还是 50 题都扣 5 分，per_file 又走"阶梯计费"两套规则
- **决策**：统一公式 base + N × perQ，管理员可配置；删除"每 10 题阶梯"逻辑
- **结果**：所见即所得；标准模式与 per_file 公式统一

### 4. 第二轮新增/解决的关键 Bug

#### Bug 4：Kimi 调用失败而 glm-4-flash 正常
- **现象**：同一份请求，DeepSeek/智谱通过、Kimi 失败
- **根因**：ChatOptions.creative() 强制 max_tokens=12000，对 Kimi 的 token 配额校验过于敏感
- **修复**：降低默认 max_tokens（creative 6000、balanced 4000）+ LlmConfig.defaultMaxTokens 改为"上限"语义（min(opts, config)）+ 4xx 错误日志增加 model/max_tokens/response_format 关键参数
- **教训**：不同厂商对 max_tokens 容忍度差异大，需按 provider 差异化默认

#### Bug 5：SourceRetrievalService 注入但未调用
- **现象**：长 PPT 仍走 distributeTruncate 盲截断
- **根因**：Phase 2 只做了 import + 字段注入，service 主流程没真正接入
- **修复**：buildSourceTextForWeek / buildPerFileSourceText / generateAndValidateQuestions 均改为先 retrieveForXxx 精筛再传 prompt
- **教训**：依赖注入不等于实际调用；重构必须验证调用链路

#### Bug 6：管理员积分硬豁免
- **现象**：管理员生成内容不扣分，与"积分流水"功能不一致
- **根因**：PointService 第 51 行硬编码 `if ("admin"...) return`
- **修复**：改为 `admin_skip_points_consume` 配置项控制，默认 false（按规则扣）；refund 同步处理避免凭空补分
- **教训**：业务规则不应硬编码，应通过配置项暴露给管理员

#### Bug 7：SSE done 事件后状态残留
- **现象**：任务完成后前端短暂显示"100% running"
- **根因**：onDone 仅依赖 getTask 异步拉数据，期间 currentTask.status 未更新
- **修复**：onDone 立即同步更新 currentTask.status / progress / stageText / resultHistoryId
- **教训**：SSE 事件 payload 已含完整字段，前端应优先用本地数据再异步兜底

#### Bug 8：TaskFloatButton 无任务档 15s 轮询导致延迟
- **现象**：用户提交任务后，右下角 badge 最长要等 15 秒才显示
- **根因**：computeInterval 无任务档 15000ms，刚提交时正处此档
- **修复**：INTERVAL_IDLE 15000 → 8000 + 监听全局 `task-submitted` CustomEvent 提交瞬间立即刷新
- **教训**：对"用户操作触发的状态变更"应用事件驱动，不应等下次 tick

### 6. RIPER-5 协议在第二轮的应用

第二轮迭代严格遵循 RIPER-5 协议（RESEARCH → INNOVATE → PLAN → EXECUTE → REVIEW）：

- **RESEARCH 阶段**：用 Glob/Grep/Read 工具系统扫描现状，输出"性能瓶颈观察"与"质量瓶颈观察"两份分析
- **INNOVATE 阶段**：对 10 个关注点各列 3-5 个备选方案，对比利弊
- **PLAN 阶段**：输出 51 个原子步骤的检查清单 + 关键文件矩阵 + 出口闸门
- **EXECUTE 阶段**：3 处微小偏差均提前报告（步骤 14 委托而非删除 / 步骤 19/20 仅挂基类不迁移 / 步骤 31 拆为 31a+31b）
- **REVIEW 阶段**：5 大维度审查（一致性 / 安全 / 性能 / 可维护性 / 兼容性），生成完整对照报告

最终结论：**实施与最终计划完全匹配，未发现未报告的偏差**。
